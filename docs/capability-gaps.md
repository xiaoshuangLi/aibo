## 1. 代码分析与语言支持

### 现状

AIBO 内置了基于 **LSP（Language Server Protocol）** 的静态代码分析能力，但当前的 LSP 客户端仅支持 `typescript-language-server`，即仅能对 **TypeScript / JavaScript** 项目提供以下增强功能：

- 悬停信息（类型、文档）
- 自动补全
- 跳转定义 / 查找引用
- 诊断信息（编译错误、类型错误）

### 不足

| 问题 | 说明 |
|------|------|
| 多语言 LSP 缺失 | Python、Go、Rust、Java、C++ 等语言的 LSP 服务器未集成，分析能力仅限文本搜索（grep）和 Bash 命令 |
| 跨语言一致性差 | 同样的"跳转定义"操作，TypeScript 项目有 LSP 加持，Python 项目只能靠 grep，体验不一致 |
| LSP 服务器自动检测 | 需要用户手动确保 `typescript-language-server` 在 PATH 中；其他语言的服务器完全未被自动检测 |

### 建议

- 按需启动对应语言的 LSP 服务器（`pylsp`、`gopls`、`rust-analyzer`、`jdtls` 等）
- 在 `lsp.ts` 中抽象语言→服务器的映射表，启动时根据项目文件类型自动选择

## 2. 多模态输入

### 现状

AIBO 仅支持**纯文本**和**语音**（中文 ASR）输入，所有工具调用、文件操作均基于文本。

### 不足

| 问题 | 说明 |
|------|------|
| 无法处理图片 | 用户无法发送截图、UI 设计稿、报错截图，AI 无法从中提取信息 |
| 无法处理 PDF / Office 文档 | 尽管有 `pdf`、`docx`、`pptx`、`xlsx` 等技能，这些技能依赖外部 CLI 工具，AI 自身不能直接读取二进制文件内容 |
| 无法处理视频 | 无录屏分析、操作回放功能 |

### 建议

- 集成支持多模态的模型（如 GPT-4o Vision、Claude 3.5 Sonnet 原生图片支持），在对话层接收 base64 或 URL 图片输入
- 为飞书机器人模式增加图片消息的解析与转发

## 3. 记忆与上下文管理

### 现状

AIBO 通过以下机制管理上下文：

- **滑动窗口记忆**（`AIBO_MEMORY_WINDOW_SIZE`，默认 5 轮）：超出窗口的历史消息被截断
- **知识库**（`knowledge_add` / `knowledge_search`）：手动存储与检索自定义知识片段
- **检查点（Checkpointer）**：支持 `memory`、`sqlite`、`filesystem` 三种存储方式，但每个线程只保存**最新一个**检查点，无历史版本

### 不足

| 问题 | 说明 |
|------|------|
| 窗口过小 | 默认 5 轮记忆对复杂编程任务严重不足，AI 经常"忘记"早期的上下文或约定 |
| 知识库为手动驱动 | 用户或 AI 需主动调用 `knowledge_add`，没有自动摘要和自动入库机制 |
| 检查点无历史版本 | 无法回溯到之前某个对话状态；出错时不能"撤回"到上一个稳定点 |
| 跨项目记忆隔离 | 不同项目切换时，上下文完全重置，无法复用跨项目的通用约定或偏好 |

### 建议

- 引入**自动压缩摘要**机制：当历史消息超过阈值时，自动调用 AI 对早期对话生成摘要并注入上下文
- 检查点支持保存多个版本，支持用户通过 `/rollback` 命令回溯
- 支持**全局用户配置文件**（类似 `.aiborc`），存储用户偏好和跨项目共享知识

## 4. 语音交互

### 现状

AIBO 支持通过腾讯云 ASR 进行语音输入，仅在**终端（Console）模式**下可用，且仅支持**中文**语音识别。

### 不足

| 问题 | 说明 |
|------|------|
| 单一 ASR 提供商 | 强依赖腾讯云，海外用户无法使用；国内用户也需额外开通腾讯云账号 |
| 无语音输出（TTS） | AI 的回复无法以语音形式播放，无法实现完整的"语音对话"体验 |
| 飞书模式不支持语音 | 飞书消息中的语音/音频附件无法被 AIBO 解析处理 |
| 仅限中文 | 无英文或其他语言的 ASR 支持 |

### 建议

- 增加对 OpenAI Whisper（`openai.audio.transcriptions`）的支持，作为腾讯云 ASR 的替代/备选
- 可选集成 TTS（如 OpenAI TTS、Azure Speech）实现语音回复
- 为飞书机器人增加语音消息的 ASR 转文字支持

## 5. 交互界面与接入渠道

### 现状

AIBO 支持两种交互模式：

1. **终端（Console）**：本地 CLI 交互，面向开发者
2. **飞书（Lark）机器人**：企业即时通讯，支持用户私聊和群聊

### 不足

| 问题 | 说明 |
|------|------|
| 无 Web UI | 没有浏览器界面，非开发者或需要远程访问时不便 |
| 无 VS Code / IDE 插件 | 无法在开发者最常用的 IDE 中直接调用 AIBO |
| 不支持 Slack / Discord / Teams | 有 `slack-gif-creator` 技能但无原生 Slack Bot 模式 |
| 飞书卡片功能有限 | 飞书互动卡片目前主要用于文字回复，不支持代码块高亮、文件预览等富交互 |
| 无 API 服务模式 | AIBO 无法以 HTTP API Server 形式运行，难以被其他系统集成调用 |

### 建议

- 增加 HTTP API Server 模式（`aibo --mode=server`），提供标准 REST / WebSocket 接口
- 开发简单的 Web Chat UI（可复用 HTTP API）
- 提供 VS Code 扩展，将 AIBO 嵌入侧边栏

## 6. 数据库与数据访问

### 现状

AIBO 没有内置的数据库工具。当前所有数据库操作都需要通过 `bash` 工具执行 CLI 命令（如 `psql`、`mysql`），要求目标数据库在本机可访问且已安装对应客户端。

### 不足

| 问题 | 说明 |
|------|------|
| 无结构化查询工具 | 无法直接执行 SQL 查询、查看表结构、浏览数据 |
| 无连接管理 | 每次操作需要用户手动提供连接字符串，没有连接池/会话复用 |
| 不支持 NoSQL | MongoDB、Redis、Elasticsearch 等无原生工具支持 |
| 安全风险 | 通过 bash 执行数据库命令时，连接凭证可能出现在命令历史中 |

### 建议

- 新增 `database` 工具集，通过 MCP 或原生实现支持 PostgreSQL、MySQL、SQLite 等
- 数据库凭证通过 `.env` 配置，不在工具调用参数中明文传递

## 7. 容器与基础设施管理

### 现状

AIBO 包含 `devops` Agent 和 `ci-cd` 技能，但底层仍依赖 `bash` 执行 Docker / Kubernetes 命令，无专用工具。

### 不足

| 问题 | 说明 |
|------|------|
| 无 Docker 工具 | 没有列出容器、查看日志、构建镜像的专用工具 |
| 无 Kubernetes 工具 | 不能直接操作 K8s 资源（Pod、Deployment、Service 等） |
| 无云平台集成 | AWS、GCP、Azure 等云服务无原生支持（需靠 Composio 或 MCP 间接实现） |

### 建议

- 考虑通过 MCP 集成 Docker Desktop API 或 kubectl 命令封装
- 在 `devops` Agent 提示词中明确声明依赖 bash 的局限性，引导用户配置正确的 MCP 工具

## 8. 代码审查与 PR 工作流

### 现状

AIBO 拥有 `code-review` 技能、`github-fetch` 工具和 `github-automation` 技能，可以从 GitHub API 获取 PR 信息并生成审查意见。

### 不足

| 问题 | 说明 |
|------|------|
| `github-fetch` 仅读 | 只能读取 GitHub 数据，不能通过原生工具创建 PR、提交 Review 评论、合并 PR |
| 依赖 Composio / MCP | 写操作需要通过 Composio 或外部 MCP Server，增加配置复杂度 |
| 无 GitLab 原生工具 | 有 `gitlab-automation` 技能，但没有对应的 GitLab API 工具；同样依赖 bash 或 MCP |
| 审查意见无结构化输出 | AI 生成的 Code Review 是自由文本，无法直接以行内注释形式提交到 PR |

### 建议

- 为 `github-fetch` 工具增加写操作（`create_pr`、`post_review_comment`、`merge_pr`）
- 提供同等的 GitLab 原生工具支持

## 9. 成本与资源管控

### 现状

AIBO 的会话管理器（`session/manager.ts`）在内部记录了每次模型调用的 Token 消耗、耗时和费用估算，但这些数据仅供调试，不对外暴露也不做限制。

### 不足

| 问题 | 说明 |
|------|------|
| 无预算限制 | 无法为单次任务或每日消耗设置 Token / 费用上限 |
| 无实时用量展示 | 用户无法在对话过程中看到已消耗的 Token 数量和估算费用 |
| 无预警机制 | Token 即将耗尽或费用超阈值时不会主动通知用户 |
| 成本数据不可导出 | 任务结束后无法导出详细的用量报告 |

### 建议

- 增加 `AIBO_MAX_TOKENS_PER_TASK` 和 `AIBO_MAX_COST_PER_DAY` 等环境变量
- 在对话结尾或达到阈值时，向用户展示本次任务的 Token 消耗摘要
- 提供用量日志导出命令（如 `aibo usage --export`）

## 10. 依赖安全与自动更新

### 现状

AIBO 没有内置的依赖漏洞扫描或自动更新能力，仅在 `ci-cd` 技能文档中建议用户在 CI 流程中集成扫描工具。

### 不足

| 问题 | 说明 |
|------|------|
| 无自动漏洞扫描 | AI 不会主动检测项目依赖是否存在已知 CVE |
| 无 Dependabot 类能力 | 无法自动创建依赖升级 PR |
| 技能文档与执行断层 | `ci-cd` 技能只给出指导，AI 无法自主运行 `npm audit` 并解读结果 |

### 建议

- 在 `devops` / `security` Agent 提示词中明确集成 `npm audit`、`pip-audit`、`trivy` 等工具
- 考虑添加 `security-scan` 工具，封装常见扫描命令并结构化输出漏洞信息

## 11. 实时协作

### 现状

每个 AIBO 实例维护一个独立的对话线程（`thread_id`），不同用户之间完全隔离。

### 不足

| 问题 | 说明 |
|------|------|
| 无多用户共享会话 | 团队成员无法加入同一个 AIBO 任务线程协同交互 |
| 飞书群聊场景受限 | 群聊模式下所有成员的消息混入同一上下文，AI 难以区分不同用户的意图和权限 |
| 无任务可见性 | 其他团队成员看不到当前正在进行的 AIBO 子任务列表和进度 |

### 建议

- 为群聊模式设计用户身份识别机制（基于飞书 `user_id`）
- 提供任务进度的只读订阅（Observe）模式，让旁观者实时查看任务进展

## 12. 可扩展性与插件机制

### 现状

AIBO 支持通过以下方式扩展：

- `agents/` 目录：添加新 Agent 的 Markdown 文件
- `skills/` 目录：添加新 Skill 的 Markdown 文件
- `mcps/` 目录：添加新 MCP 工具的 JSON 配置文件

### 不足

| 问题 | 说明 |
|------|------|
| 无插件 API / SDK | 第三方开发者只能通过 Fork 仓库来添加原生 TypeScript 工具，没有稳定的插件 API |
| MCP 无自动发现 | MCP 工具需要手动创建 JSON 配置，无法从 npm 包或标准目录自动发现已安装的 MCP 服务 |
| Skills / Agents 仅文档驱动 | Skills 和 Agents 是纯 Markdown 提示词，无法包含可执行逻辑，能力上限受限于 AI 理解能力 |
| 无版本管理 | Skills 和 Agents 的 Markdown 文件没有版本号，多人维护时易产生冲突 |

### 建议

- 发布 `@boay/aibo-sdk`，提供标准的工具注册接口（`registerTool`、`registerAgent`）
- 支持从 `node_modules` 自动发现带有 `aibo-plugin` 关键词的 npm 包

## 汇总优先级

| 优先级 | 能力项 | 理由 |
|--------|--------|------|
| 🔴 高 | 多语言 LSP 支持 | 编程助手的核心能力，非 TypeScript 项目体验差距明显 |
| 🔴 高 | 上下文记忆增强（自动摘要、多版本检查点） | 复杂任务中频繁丢失上下文是最常见的用户痛点 |
| 🔴 高 | GitHub 写操作工具 | 当前依赖 Composio，配置门槛高 |
| 🟡 中 | 多模态图片输入 | 随着 Vision 模型普及，用户期望越来越高 |
| 🟡 中 | HTTP API Server 模式 | 便于与 CI/CD、其他系统集成 |
| 🟡 中 | Token / 费用管控 | 大模型成本敏感场景中不可缺少 |
| 🟡 中 | 多 ASR 提供商支持 | 降低海外用户和英文用户的使用门槛 |
| 🟢 低 | Web UI | 可优先通过 HTTP API 模式让社区实现 |
| 🟢 低 | 数据库工具 | 可通过 MCP 间接实现，优先级低于核心编程能力 |
| 🟢 低 | 插件 SDK | 生态建设的长期目标 |
